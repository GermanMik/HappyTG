# Evidence

## Root cause

- The local host-run BaseDeploy Caddy process was not serving the HappyTG route at first because port `3002` was occupied by a temporary Mini App smoke-test server.
- After Caddy was started, the live BaseDeploy `Caddyfile` had drifted and no longer routed `happytg.gerta.crazedns.ru` to HappyTG, so the domain served the wrong app.
- The no-port `https://happytg.gerta.crazedns.ru/miniapp` route became reachable by `curl`, but Chromium/WebView still rejected it with `ERR_SSL_PROTOCOL_ERROR`.
- The public `https://happytg.gerta.crazedns.ru:5083/miniapp` route opens successfully in Chromium and is accepted by HappyTG validation.

## Changes made

- Restored HappyTG routing in `C:\Develop\Projects\BaseDeploy\caddy\Caddyfile`.
- Added `C:\Develop\Projects\BaseDeploy\docs\happytg-caddy-route-repair-2026-06-10.md`.
- Updated local runtime `.env` so `HAPPYTG_MINIAPP_URL=https://happytg.gerta.crazedns.ru:5083/miniapp`.
- Recreated the HappyTG Docker services with `docker compose --env-file .env -f infra\docker-compose.example.yml up -d ...`.
- Ran `pnpm happytg telegram menu set`; Telegram Bot API accepted the menu button with the `:5083` Mini App URL.
