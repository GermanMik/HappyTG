# HappyTG 0.4.5

HappyTG `0.4.5` closes the remaining Telegram Mini App launch and dashboard routing gaps without widening the public API surface.

## Highlights

- `0.4.5`
- Telegram menu setup, doctor, and verify now require the public `/miniapp` route to prove HappyTG Mini App identity instead of treating any `HTTP 200 text/html` response as healthy.
- Wrong-product public HTML, such as HealthOS content served at the HappyTG Mini App URL, now fails launch preflight with a useful body detail.
- The public Caddy contract now includes the exact authenticated dashboard route, `/api/v1/miniapp/dashboard`, so same-origin Mini App dashboard requests reach API session auth.
- Generic public `/api/*` remains denied; no broad `/api/v1/miniapp*` wildcard was introduced.

## Upgrade Notes

- Keep `HAPPYTG_MINIAPP_URL` pointed at the public HTTPS `/miniapp` route:

  ```env
  HAPPYTG_MINIAPP_URL=https://<domain>/miniapp
  ```

- Self-hosted Caddy deployments should carry the explicit Mini App public API allowlist:

  ```caddyfile
  handle /api/v1/miniapp/auth/session {
    reverse_proxy api:4000
  }

  handle /api/v1/miniapp/dashboard {
    reverse_proxy api:4000
  }

  @miniappApprovalResolve path_regexp miniapp_approval_resolve ^/api/v1/miniapp/approvals/[^/]+/resolve$
  handle @miniappApprovalResolve {
    reverse_proxy api:4000
  }

  handle /api/* {
    respond "Not found" 404
  }
  ```

- Unauthenticated dashboard requests should return `401 Mini App session auth required`; a `404` at that route means Caddy is still blocking it before API auth.
