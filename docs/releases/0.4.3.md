# HappyTG 0.4.3

HappyTG `0.4.3` adds an explicit Docker Compose startup path to the installer and rolls up the Telegram Mini App launch and Windows polling fallback fixes merged after `0.4.2`.

## Highlights

- `0.4.3`
- `pnpm happytg install` now lets the operator choose a startup mode explicitly: local `pnpm dev`, packaged Docker Compose startup, manual commands, or skip.
- Docker installer startup validates `docker compose`, runs the packaged control-plane stack, reports Compose and host readiness, and still tells the operator to pair/start the host daemon outside Docker.
- Docker Compose startup guidance now uses `docker compose --env-file .env -f infra/docker-compose.example.yml ...`, so saved `HAPPYTG_*_PORT` overrides affect published host ports as expected.
- Telegram Mini App menu launch now uses the public HTTPS `/miniapp` route and keeps a Windows PowerShell fallback when Telegram API access through Node fetch is unavailable.
- Windows polling fallback switches to PowerShell `getUpdates` faster after Node transport timeout, reducing delayed bot startup on affected hosts.
- Local launch diagnostics now point operators at the local Mini App URL/port instead of misdirecting them to the API URL.

## Upgrade Notes

- For installer-driven Docker startup, use the same command family the installer now prints and executes:

  ```bash
  docker compose --env-file .env -f infra/docker-compose.example.yml up --build -d
  ```

- Docker startup still covers only the packaged control plane. The host daemon remains host-run because it needs access to local repositories and local Codex configuration:

  ```bash
  pnpm daemon:pair
  pnpm dev:daemon
  ```

- Local development is still the default dev path and stays separate from full Compose startup:

  ```bash
  pnpm dev
  ```

- Telegram Mini App buttons and the persistent menu still require a public HTTPS HappyTG URL with the `/miniapp` route available to Telegram clients.
