# HappyTG 0.4.1

HappyTG `0.4.1` repairs Telegram Mini App launch behavior across local development and production-style deployments.

## Highlights

- `0.4.1`
- Local Telegram bot replies remain functional when Mini App URLs resolve to HTTP, localhost, private-network, or malformed values; invalid `web_app` buttons are omitted instead of causing Telegram `sendMessage` HTTP 400 failures.
- Valid public HTTPS Mini App URLs still render inline `web_app` buttons for `/start`, `/menu`, sessions, approvals, reports, diff, and verify flows.
- Windows PowerShell `sendMessage` fallback diagnostics now include Telegram's JSON rejection description when Telegram returns one.
- Persistent Telegram menu-button setup is now an explicit operator command: `pnpm happytg telegram menu set`, with `--dry-run`, `reset`, public HTTPS URL validation, and public Caddy `/miniapp` preflight.
- Caddy and docs keep the production public contract aligned: `/miniapp`, allowed Mini App API endpoints, and `/telegram/webhook`.

## Upgrade Notes

- Local `.env` defaults may keep `HAPPYTG_PUBLIC_URL=http://localhost:4000` and `HAPPYTG_APP_URL=http://localhost:3001`; Telegram Web App buttons will be disabled until a public HTTPS Mini App URL is configured.
- For production, set `HAPPYTG_MINIAPP_URL=https://<domain>/miniapp` or a public HTTPS `HAPPYTG_PUBLIC_URL` that serves `/miniapp`.
- Configure the persistent Telegram menu button explicitly after public routing works:

  ```bash
  pnpm happytg telegram menu set --dry-run
  pnpm happytg telegram menu set
  ```
