# HappyTG 0.4.4

HappyTG `0.4.4` repairs the Telegram Mini App interaction failure on the public HTTPS `/miniapp` route and tightens the Mini App into a clearer mobile-first control surface without changing the core architecture.

## Highlights

- `0.4.4`
- Public reverse-proxied Mini App pages no longer inject `http://localhost:4000` into `window.HAPPYTgApiBase` when the browser is actually running on `https://<domain>/miniapp`.
- With `HAPPYTG_BROWSER_API_URL` left empty, public HTTPS `/miniapp` requests now use same-origin `/api/...`, while direct localhost Mini App development still falls back to the configured local API origin.
- Auth bootstrap, approval actions, and new-task submission now expose visible loading, success, and failure feedback instead of silently failing.
- The Mini App shell now uses a stronger mobile hierarchy: active bottom navigation, larger touch targets, clearer CTA placement, and explicit recovery states for auth and action failures.

## Upgrade Notes

- Keep the public Telegram Mini App URL on the HTTPS `/miniapp` route:

  ```env
  HAPPYTG_MINIAPP_URL=https://<domain>/miniapp
  ```

- For production or self-hosted Caddy, leave `HAPPYTG_BROWSER_API_URL` empty so browser-side Mini App requests stay same-origin on the public origin:

  ```env
  HAPPYTG_BROWSER_API_URL=
  ```

- Direct localhost development still works with the local API origin. If you run the Mini App directly on a host port such as `3007`, keep `HAPPYTG_APP_URL`, `HAPPYTG_MINIAPP_PORT`, and `HAPPYTG_DEV_CORS_ORIGINS` aligned.

- The public Caddy contract is unchanged: keep only the documented Mini App auth/session and approval-action endpoints public. Do not expose generic `/api/*` just to satisfy browser routing.
