# HappyTG 0.4.19

HappyTG `0.4.19` releases a reusable Mini App speed optimization prompt for future proof-backed performance work.

## Highlights

- `0.4.19`
- Added `docs/prompts/happytg-miniapp-speed-optimization.md` as a repo-local execution prompt for measuring and improving Telegram Mini App speed.
- The prompt requires baseline route timings, API timings, payload sizes, slow-dependency simulations, and after-change evidence before any performance claim.
- Added a 10-role critical review model so future speed work is checked from Mini App operator, WebView performance, backend/API, Codex Desktop adapter, control-plane safety, security/privacy, cache correctness, QA, release operations, and Graphify/navigation perspectives.
- The workflow preserves HappyTG invariants for Telegram transport boundaries, policy/approval order, serialized mutations, source discrimination, session/auth behavior, and local LM Studio assumptions.

## Safety Notes

- This release adds documentation and proof artifacts only; it does not change production runtime behavior.
- The prompt explicitly forbids weakening auth, policy, approval, serialized mutation, public Mini App URL, or Telegram `initData` contracts for a speed pass.
- No migration is required.

- `pnpm release:check --version 0.4.19`
- Prompt artifact review against HappyTG architecture invariants and existing proof-loop/release guidance.
