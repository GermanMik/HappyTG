# HappyTG 0.3.18

## Release version

- `0.3.18`

## Why this release

- Fix the broken local Telegram bot UX where `pnpm dev` could look healthy while `/start` and `/pair <CODE>` never reached the bot without a public webhook.
- Make Telegram delivery mode explicit and honest so local development can use polling while stable/self-hosted deployments stay webhook-first.
- Synchronize docs and proof artifacts with the actual bot runtime behavior.

## Fixed

- `apps/bot/src/index.ts` now supports deterministic Telegram delivery mode selection with `auto`, `polling`, and `webhook`, and local/non-public `HAPPYTG_PUBLIC_URL` values now route to Telegram polling instead of leaving the bot webhook-only.
- Local polling now receives `/start` and `/pair <CODE>` through the same bot handlers used by webhook delivery, preserving the existing `/api/v1/pairing/claim` boundary and avoiding duplicate mixed-mode processing.
- Bot startup and `/ready` now report honest Telegram delivery state, including degraded webhook diagnostics when webhook mode is selected but Telegram is not pointed at the expected public webhook URL.

## Changed

- `.env.example`, `docs/configuration.md`, `docs/quickstart.md`, `docs/installation.md`, and `docs/self-hosting.md` now describe `TELEGRAM_UPDATES_MODE=auto|polling|webhook`, local polling defaults, and webhook-first stable deployment expectations.
- The canonical source proof bundle for the bot fix is now complete and validated, and this release adds a dedicated publish bundle for the `0.3.18` metadata/update path.
