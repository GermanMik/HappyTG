# HappyTG 0.3.4

## Release version

- `0.3.4`

## Why this release

- Close the remaining Windows follow-up after `0.3.3`, where Codex could still be installed and runnable through the standard `%APPDATA%\\npm` shim path but bootstrap would classify it as missing from the current shell.
- Keep the existing bootstrap and installer architecture intact while making the Windows PATH diagnosis more precise and less alarming.

## Fixed

- Bootstrap now searches standard Windows user npm bin directories such as `%APPDATA%\\npm` for Codex wrappers when direct `npm prefix -g` probing is unavailable.
- Runnable Codex APPDATA wrapper cases now resolve to `CODEX_PATH_PENDING` warnings instead of cascading into missing-Codex failures during installer post-checks.
- PATH follow-up diagnostics now show the recovered wrapper directory directly, so the user sees the exact path that needs to be added to `PATH`.
- Installer runtime coverage now locks this scenario at `success-with-warnings` rather than `recoverable-failure`.

## Verification

- Added regression coverage for:
  - Windows APPDATA/npm Codex wrapper detection without npm prefix probing
  - installer post-check summaries that run the real bootstrap checks through that Windows wrapper path
- Release validated with:
  - `pnpm --filter @happytg/bootstrap test`
  - `pnpm build`
  - `pnpm typecheck`
  - `pnpm lint`
  - `pnpm release:check --version 0.3.4`
  - `pnpm happytg task validate --repo . --task HTG-2026-04-13-windows-codex-appdata-wrapper`

## Upgrade notes

- Update all workspace versions to `0.3.4`.
- If Windows bootstrap reports `CODEX_PATH_PENDING` and points at `%APPDATA%\\npm`, treat it as a shell PATH follow-up rather than a missing Codex install.
