# HappyTG 0.3.12

## Release version

- `0.3.12`

## Why this release

- Replace the noisy flat install `nextSteps` list with a structured automation model that can distinguish completed local work from real manual handoff, warnings, reuse guidance, and conflicts.
- Auto-run the safe local install-finalization work that should not be left to the user, while keeping explicit Telegram/user boundaries truthful.
- Make install completion output consistent across JSON, plain text, and TUI without hiding real environment constraints on the maintainer machine.

## Fixed

- Install finalization is now built from a shared structured model instead of ad-hoc string concatenation. Bootstrap onboarding and install final summaries now classify follow-up as `auto`, `manual`, `warning`, `reuse`, `conflict`, or `blocked`.
- The installer can auto-request a host pairing code when the API and Telegram prerequisites are satisfied, then hand the user a concrete `/pair CODE` command exactly once. When Telegram validation fails, pairing is blocked explicitly instead of still suggesting `pnpm daemon:pair`.
- Background launcher messaging now reflects the actual configuration result. If an automated mode falls back to manual, the final output reports that fallback as a warning instead of falsely presenting it as already configured.
- Reuse guidance and conflicts are deduplicated across setup/doctor/verify/install flows. Shared infra reuse, running-stack reuse, mini app port conflict guidance, and pending pair/daemon work no longer collide with contradictory `pnpm dev` or duplicate warning text.
- Plain-text and TUI summaries now render grouped sections for completed auto-run steps, manual handoff, blocked prerequisites, reuse guidance, conflicts, and warnings, while JSON exposes the same structured `finalization.items` / `onboarding.items` model.
