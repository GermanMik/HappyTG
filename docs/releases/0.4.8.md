# HappyTG 0.4.8

HappyTG `0.4.8` repairs installer Docker reuse, host-daemon startup guidance, and Windows bootstrap warning UX.

## Highlights

- `0.4.8`
- The Windows one-line installer demotes successful temporary bootstrap ignored-build-script output to a short informational line after the `tsx`/`esbuild` preflight marker passes.
- Docker install asks whether to reuse system Redis/Postgres/MinIO/Caddy or run an isolated Docker stack.
- Docker reuse mode avoids duplicate infra/Caddy containers and passes container-reachable Compose endpoints into app services.
- System Caddy defaults to snippet generation; patching is guarded by confirmation, backup, validate-before-reload, and a marked HappyTG-managed block.
- Host-daemon startup is clearly separate from Docker Compose, and final output includes inspect/log/restart/stop Docker commands.
- Install-time background reset removes stale HappyTG-owned launchers before creating the newly selected Scheduled Task/Startup/manual/skip state.

## Upgrade Notes

- No data migration is required.
- Non-interactive Docker install keeps isolated mode unless `--docker-services reuse` is passed explicitly.
- If using system Caddy, review the generated snippet before applying it to operator-owned Caddy config.
- The host daemon remains outside Docker Compose and must still be paired/started on the host when host operations are needed.
