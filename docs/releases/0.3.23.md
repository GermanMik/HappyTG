# HappyTG 0.3.23

## Release version

- `0.3.23`

## Why this release

- Ship the already-landed worker/bot startup fix as a user-visible release instead of leaving it only on `main`.
- Make `pnpm dev` truthful when `4100` or `4200` is already occupied by an existing HappyTG service.
- Publish the validated startup-port proof from `HTG-2026-04-19-startup-port-proof-loop`.

## Fixed

- `apps/worker/src/index.ts` now classifies occupied worker ports into same-service reuse, different-HappyTG-service conflict, and foreign-listener conflict instead of crashing with a raw Node `EADDRINUSE` stack trace.
- Worker runtime startup now waits until the probe server actually owns the port before starting local maintenance ticks, so same-service reuse does not start duplicate worker activity.
- `apps/bot/src/index.ts` now follows the same reuse-vs-conflict startup model for occupied port `4100`, replacing the old `Bot failed to start` + raw `listen EADDRINUSE` detail on expected occupied-port paths.
- Bot reuse no longer starts a second polling/webhook lifecycle when a HappyTG Bot is already listening on the target port.
- `apps/bot/src/index.test.ts` and `apps/worker/src/index.test.ts` now lock regression coverage for free-port bind, same-service reuse, different-service conflict, and foreign HTTP listener conflict.

## Changed

- `docs/quickstart.md` and `docs/troubleshooting.md` now describe truthful occupied-port handling for bot `4100` and worker `4200`, alongside the existing API and Mini App guidance.
- The canonical source proof bundle for this release is `HTG-2026-04-19-startup-port-proof-loop`, and this publication adds a dedicated release bundle for the `0.3.23` metadata/publish path.
