# HappyTG 0.3.15

## Release version

- `0.3.15`

## Why this release

- Close the remaining gap between truthful setup/bootstrap port diagnostics and the interactive installer UX.
- Stop treating real port conflicts as passive follow-up during install when the product can ask the user for an explicit choice immediately.
- Keep occupied-port handling honest across setup, installer, and runtime paths without weakening env override precedence or supported reuse semantics.

## Fixed

- `packages/bootstrap/src/index.ts` now keeps three nearby non-colliding free-port suggestions for planned-port conflicts while preserving service-specific `HAPPYTG_*_PORT` precedence and `PORT` fallback for app services.
- `packages/bootstrap/src/install/index.ts` and `packages/bootstrap/src/install/tui.ts` now run an installer-native planned-port preflight before later startup guidance, classify supported reuse vs real conflict, show the detected listener when available, and let the user pick a suggested port, enter a custom one, or abort.
- `packages/bootstrap/src/index.test.ts` and `packages/bootstrap/src/install.runtime.test.ts` now lock regression coverage for suggested-port lists, supported reuse, interactive suggested-port selection, manual entry, and explicit refusal to continue.
- `apps/api/src/index.ts`, `apps/miniapp/src/index.ts`, and the related tests/docs in this release train keep runtime occupied-port messaging aligned with the same reuse-vs-conflict model used by bootstrap and installer flows.
