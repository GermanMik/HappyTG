# HappyTG 0.3.16

## Release version

- `0.3.16`

## Why this release

- Remove the last dishonest installer pairing branch where any existing local host state forced the user back to manual `pnpm daemon:pair`.
- Preserve the real Telegram `/pair <CODE>` claim boundary while automating the safe parts of install-time pairing.
- Make installer finalization truthful for existing hosts: reuse already paired hosts, refresh only still-registering hosts, and keep blocked/manual fallback explicit when the backend probe cannot complete.

## Fixed

- `packages/bootstrap/src/install/pairing.ts` adds an installer-facing pairing decision state machine that reads local daemon state, probes backend host status, and classifies `reuse-existing-host`, `auto-requested`, and `manual-fallback` without moving the Telegram claim boundary.
- `packages/bootstrap/src/install/index.ts` now routes finalization through that state machine, so an existing `active` or `paired` host stays on reuse, an existing `registering` host auto-refreshes the pairing code during install, and blocked pairing no longer emits contradictory daemon-start guidance.
- `packages/bootstrap/src/install.runtime.test.ts` now locks deterministic regression coverage for all critical pairing branches: no local host, existing `registering`, existing `active`, invalid token, and existing-host backend probe unavailable.
- `packages/bootstrap/src/install/tui.ts` and `packages/bootstrap/src/install.test.ts` now keep the final installer summary close path honest on real TTYs by resuming stdin while waiting for `Enter` and pausing it again during cleanup, which lets the command return to the shell prompt.

## Changed

- `apps/api/src/index.test.ts` now gives the transient API handoff regression a wider deterministic timing window on slower Windows hosts, keeping the guarded release `pnpm test` gate stable without changing production startup behavior.
