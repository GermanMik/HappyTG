# HappyTG 0.3.10

## Release version

- `0.3.10`

## Why this release

- Tighten the remaining installer diagnostics wording and summary behavior after `0.3.9`, without hiding legitimate environment warnings.
- Remove the last contradictory `install` next-step path where HappyTG could tell the user to reuse an already-running stack and still suggest `pnpm dev` in the same final summary.

## Fixed

- Telegram bot `getMe` warnings now explicitly describe the failure as Node HTTPS/undici transport-specific when a Windows PowerShell Bot API probe validates the same token, and the guidance now points to concrete proxy env vars plus IPv4/IPv6 routing checks instead of a vague "Node or curl" hint.
- Final installer next steps now remove contradictory `pnpm dev` guidance when post-checks already detected running HappyTG services, and overlapping shared-infra / Redis / running-stack advice is semantically deduplicated into one concise path.
- The interactive installer no longer re-renders the exact same `Final Summary` screen again on `ENTER`, so transcript/log captures stop showing a duplicated final screen artifact.

  - `pnpm lint`

  - `pnpm release:check --version 0.3.10`
  - `pnpm happytg task validate --repo . --task HTG-2026-04-17-release-0310-warning-triage`

## Upgrade notes

- Update all workspace versions to `0.3.10`.
- On the maintainer machine used for verification, the remaining direct warning set is still legitimate by design: Telegram Bot API over Node HTTPS times out while a Windows PowerShell Bot API probe succeeds, Codex websocket startup returns `403 Forbidden` before the CLI falls back to HTTP, and Mini App port `3001` is occupied by `contacts-frontend`.
- `pnpm happytg install --json` without `--dirty-worktree keep` still stops early on a dirty checkout by design; the scoped summary/next-step verification path was captured with the explicit keep strategy so installer post-check output could run against the current workspace.
