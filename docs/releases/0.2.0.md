# HappyTG 0.2.0

## Release version

- `0.2.0`

## Why this release

- Fix the broken Windows bootstrap path that reported `Codex CLI not found` even when `codex --version` worked.
- Fix the Windows home-resolution regression in `@happytg/shared`.
- Turn first start into an explicit, guided flow for Telegram bot token setup, Redis checks, port conflicts, and pairing.

## Fixed

- Windows-safe `resolveHome()` now respects `HOME`, `USERPROFILE`, `HOMEDRIVE` + `HOMEPATH`, and keeps Unix/macOS behavior intact.
- Codex runtime detection now handles Windows `Path` / `PATH`, `PATHEXT`, and `codex.cmd` shim scenarios without false negatives in bootstrap checks.
- Runtime entrypoints now load `.env` automatically for local first start instead of silently behaving as if required variables were unset.

## Improved onboarding

- Added `pnpm happytg setup` as the compact first-run checklist.
- Telegram bot token setup is now explicit, validates obvious missing/invalid states, and never prints secrets.
- Pairing handoff is shorter: token -> shared infra -> `pnpm dev` -> `pnpm daemon:pair` -> `/pair <CODE>` -> `pnpm dev:daemon`.

## Redis and port handling

- Bootstrap now distinguishes Redis absent, installed-but-stopped, running, and non-Redis port conflicts.
- Critical ports are checked up front, including `3001`, `4000`, `4100`, `4200`, and `6379`.
- Port results distinguish free, occupied by the expected HappyTG service, and occupied by an external process or system service.
- Alternative port selection is now documented and supported through service-specific env vars such as `HAPPYTG_MINIAPP_PORT` and `HAPPYTG_REDIS_HOST_PORT`.
- Compose guidance now explains how to reuse an existing system Redis instead of failing on `6379`.

## Docs

- Updated `README.md`, `docs/quickstart.md`, `docs/installation.md`, `docs/bootstrap-doctor.md`, `docs/configuration.md`, `docs/troubleshooting.md`, and `infra/README.md`.
- Added Windows PowerShell equivalents for `.env` creation and port override commands.
- Clarified that local development should use compose for shared infra and `pnpm dev` for repo services, not both app stacks on the same default ports.

## Verification

- Added regression coverage for Windows home resolution.
- Added regression coverage for Windows-like Codex shim detection.
- Added bootstrap tests for token onboarding, Redis states, and port conflict messaging.
- Release validated with the standard local gates: `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm build`, `pnpm happytg doctor`, and `pnpm happytg verify`.

## Upgrade notes

- Update all workspace versions to `0.2.0`.
- Refresh `.env` from the new `.env.example` if you want the new service-specific port variables.
- If you already run Redis on `localhost:6379`, reuse it and skip compose `redis` for the local `pnpm dev` path.
- If you previously used one-shot `PORT=...` overrides, they still work for individual services, but the recommended stable path is the new service-specific env vars.
