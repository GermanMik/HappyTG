# HappyTG 0.3.8

## Release version

- `0.3.8`

## Why this release

- Ship the merged installer diagnostics follow-up that separated real product bugs from legitimate local-environment warnings in `setup`, `doctor`, and `verify`.
- Make first-run diagnostics more actionable on Windows and local Docker-heavy setups by classifying Telegram lookup failures, Codex wrapper/smoke behavior, and planned port reuse vs conflict before services start.

## Fixed

- Telegram bot `getMe` checks now explain whether the failure is due to DNS, timeout, TLS/certificate, proxy, socket, HTTP, invalid-token, or non-JSON response issues instead of reporting only `fetch failed`.
- Windows Codex readiness checks now prefer runnable wrapper companions such as `codex.cmd`, preserve smoke prompts with spaces, close stdin for non-interactive runs, and summarize only actionable stderr root causes.
- Planned-port diagnostics now analyze the full current HappyTG port plan, attribute active listeners, classify supported Redis/Postgres/MinIO reuse separately from third-party conflicts, and suggest safe non-colliding overrides for conflicting ports.
- Final summaries no longer contradict per-check findings by reporting a narrower port subset than the actual warning surface.

## Verification

  - `pnpm typecheck`
  - `pnpm test`
  - `pnpm release:check --version 0.3.8`

## Upgrade notes

- Update all workspace versions to `0.3.8`.
- On the maintainer machine used for verification, the only remaining installer warnings after these fixes were a real Mini App port conflict on `3001` and a real Codex websocket `403 Forbidden` environment warning; supported local Redis/Postgres/MinIO listeners are now treated as reuse, not failures.
